## UX / QA Lessons

- Для расчётных дефектов нельзя ограничиваться проверкой наличия ЭК в seed.
  Нужно прогонять реальный JSON пользователя через полный load pipeline
  (`prepareLoadedCalc`), затем смотреть фактические qty/cost по каждому
  применимому стенду.
- `null`, `undefined`, числовые строки и значения вне `min/max` в ответах JSON
  не должны молча превращаться в `0`. При наличии `defaultIfUnknown` /
  `defaultValue` загрузка обязана либо автоматически нормализовать значение с
  отчётом пользователю, либо дать ручной путь проверки через Опросник/реестр
  допущений.
- Новый расчёт не должен заполнять числовые вопросы искусственным нулём, если
  в модели вопроса задан `defaultIfUnknown`. Иначе “не знаю” становится
  валидным нулём и обнуляет RAM, CPU, SSD/HDD/S3 или другие связанные ЭК.
- При аудите инфраструктурных ЭК проверять не только синтаксис формул, но и
  связность модели: активный стенд с вычислениями не может остаться без CPU,
  RAM и рабочего SSD; БД должна давать SSD, файлы — объектное хранилище, а
  бэкапы на стендах, где модель предусматривает HDD, должны давать HDD.
- Не придумывать внешние “нормы” без источника. Инварианты расчёта формулировать
  только из текущей модели проекта (applicableStands, `min/max`,
  `defaultIfUnknown`, формулы и единицы ЭК) либо из проверенного внешнего
  источника с явной ссылкой.
- Для багов PDF/печати нельзя закрывать задачу только по DOM, computed styles,
  unit-тестам или факту, что `page.pdf()` создал файл. Нужно воспроизвести
  пользовательский режим и проверить реальный PDF как картинку: PDF -> PNG
  или эквивалентный raster/screenshot артефакт, где видны строки и значения.
- Проверять ровно тот режим, где пользователь показал дефект: тема, вкладка,
  подвкладка, фильтр `Скрыть без бюджета`, отключённые стенды, risk-флаг и
  выбранный сценарий печати. Если баг был в тёмной теме PDF, проверка только
  светлой темы не засчитывается.
- Для `Сводка AI-метрик` фильтр `Скрыть без бюджета` должен ориентироваться на
  фактически видимые значения строки на активных стендах и в `ИТОГО`. Строки,
  где на экране одни `—` или `0`, например `Токены` и `CPU агентов` при
  нулевых запросах/выключенных агентах, должны скрываться. Нельзя подменять это
  косвенной проверкой цены или исходной ЭК-строки, если UI всё равно показывает
  пустую метрику.
- Для таблиц в PDF проверять не только заголовок и наличие блока, а строки
  `tbody`: значения должны быть визуально читаемы на белом фоне print-режима.
- После пользовательского скриншота с регрессией обязательно сохранять
  проверочные артефакты (`.pdf`, `.png`/crop) и сверять их глазами или
  пиксельным guard'ом перед финальным ответом.
- PDF Опросника состоит из нескольких таблиц, но для пользователя это один
  документ с единой сеткой `Вопрос / Ответ / Пояснение`. Отдельные таблицы
  вроде `Параметры расчёта` обязаны явно наследовать ширины колонок основной
  таблицы; проверять это координатами ячеек в Playwright, иначе браузерский
  `table-layout: auto` может сместить значения влево только в print/PDF.
- Если блок PDF Опросника визуально продолжает одну таблицу, не выносить его в
  отдельный `<table>` даже с похожим `colgroup`: в extended-режиме Chromium PDF
  всё равно может пересчитать layout иначе. `Параметры расчёта` должны быть
  `tbody` той же таблицы, а координатный guard обязан проверять и компактный, и
  extended-режим с колонкой `Пояснение`.
- Quick Start не должен создавать `answers` без вопроса в seed: такие поля
  становятся фантомными, пользователь думает, что они участвуют в расчёте, а
  формулы их не читают. Для каждого wizard-output id нужен seed question,
  формульное использование или явный служебный allowlist с тестом.
- При изменении формул уже существующих ЭК учитывать legacy JSON: сохранённый
  расчёт несёт собственный `dictionaries.items`, поэтому новый seed не
  применится сам. Такие ЭК надо добавлять в refresh-list load pipeline и
  проверять реальный импортированный JSON через `prepareLoadedCalc`.
