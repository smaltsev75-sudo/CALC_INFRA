## UX / QA Lessons

- Для расчётных дефектов нельзя ограничиваться проверкой наличия ЭК в seed.
  Нужно прогонять реальный JSON пользователя через полный load pipeline
  (`prepareLoadedCalc`), затем смотреть фактические qty/cost по каждому
  применимому стенду.
- `null`, `undefined`, числовые строки и значения вне `min/max` в ответах JSON
  не должны молча превращаться в `0`. При наличии `defaultIfUnknown` /
  `defaultValue` загрузка обязана либо автоматически нормализовать значение с
  отчётом пользователю, либо дать ручной путь проверки через Опросник/реестр
  допущений.
- Новый расчёт не должен заполнять числовые вопросы искусственным нулём, если
  в модели вопроса задан `defaultIfUnknown`. Иначе “не знаю” становится
  валидным нулём и обнуляет RAM, CPU, SSD/HDD/S3 или другие связанные ЭК.
- При аудите инфраструктурных ЭК проверять не только синтаксис формул, но и
  связность модели: активный стенд с вычислениями не может остаться без CPU,
  RAM и рабочего SSD; БД должна давать SSD, файлы — объектное хранилище, а
  бэкапы на стендах, где модель предусматривает HDD, должны давать HDD.
- Не придумывать внешние “нормы” без источника. Инварианты расчёта формулировать
  только из текущей модели проекта (applicableStands, `min/max`,
  `defaultIfUnknown`, формулы и единицы ЭК) либо из проверенного внешнего
  источника с явной ссылкой.
- Для багов PDF/печати нельзя закрывать задачу только по DOM, computed styles,
  unit-тестам или факту, что `page.pdf()` создал файл. Нужно воспроизвести
  пользовательский режим и проверить реальный PDF как картинку: PDF -> PNG
  или эквивалентный raster/screenshot артефакт, где видны строки и значения.
- Проверять ровно тот режим, где пользователь показал дефект: тема, вкладка,
  подвкладка, фильтр `Скрыть без бюджета`, отключённые стенды, risk-флаг и
  выбранный сценарий печати. Если баг был в тёмной теме PDF, проверка только
  светлой темы не засчитывается.
- Для `Сводка AI-метрик` фильтр `Скрыть без бюджета` должен ориентироваться на
  фактически видимые значения строки на активных стендах и в `ИТОГО`. Строки,
  где на экране одни `—` или `0`, например `Токены` и `CPU агентов` при
  нулевых запросах/выключенных агентах, должны скрываться. Нельзя подменять это
  косвенной проверкой цены или исходной ЭК-строки, если UI всё равно показывает
  пустую метрику.
- Для таблиц в PDF проверять не только заголовок и наличие блока, а строки
  `tbody`: значения должны быть визуально читаемы на белом фоне print-режима.
- После пользовательского скриншота с регрессией обязательно сохранять
  проверочные артефакты (`.pdf`, `.png`/crop) и сверять их глазами или
  пиксельным guard'ом перед финальным ответом.
- PDF Опросника состоит из нескольких таблиц, но для пользователя это один
  документ с единой сеткой `Вопрос / Ответ / Пояснение`. Отдельные таблицы
  вроде `Параметры расчёта` обязаны явно наследовать ширины колонок основной
  таблицы; проверять это координатами ячеек в Playwright, иначе браузерский
  `table-layout: auto` может сместить значения влево только в print/PDF.
- Если блок PDF Опросника визуально продолжает одну таблицу, не выносить его в
  отдельный `<table>` даже с похожим `colgroup`: в extended-режиме Chromium PDF
  всё равно может пересчитать layout иначе. `Параметры расчёта` должны быть
  `tbody` той же таблицы, а координатный guard обязан проверять и компактный, и
  extended-режим с колонкой `Пояснение`.
- Quick Start не должен создавать `answers` без вопроса в seed: такие поля
  становятся фантомными, пользователь думает, что они участвуют в расчёте, а
  формулы их не читают. Для каждого wizard-output id нужен seed question,
  формульное использование или явный служебный allowlist с тестом.
- При изменении формул уже существующих ЭК учитывать legacy JSON: сохранённый
  расчёт несёт собственный `dictionaries.items`, поэтому новый seed не
  применится сам. Такие ЭК надо добавлять в refresh-list load pipeline и
  проверять реальный импортированный JSON через `prepareLoadedCalc`.
- Активный вопрос Опросника не должен быть декоративным по умолчанию. Каждый
  вопрос обязан либо напрямую/косвенно влиять на ЭК, бюджетный контроль или
  плановый контекст, либо иметь явный пользовательский статус «Информационное
  поле» и архитектурный тест покрытия.
- Для Dashboard/Details проверять не только наличие рассчитанного ЭК в
  `calculate()`, но и фактическую видимость агрегата в пользовательском UI на
  импортированном JSON. Малые значения хранилищ в ТБ нельзя округлять до нуля:
  если стенд получил `0.12 ТБ`, пользователь должен видеть `0,12 ТБ`, а не
  `—`. Токены AI-нагрузки проверять отдельным e2e-контрактом в Dashboard,
  Details и Quick Start.
- Сходимость Dashboard и Детализации обязательна: суммы по активным стендам,
  категории, строка ИТОГО и qty-агрегаты должны браться из одного расчёта и
  одной семантики. Если Dashboard показывает qty с capacity-буферами при
  включённых рисках, Детализация → Объём (qty) обязана показывать тот же
  режим, а не сырой qty. Этот контракт проверять e2e после Quick Start,
  изменения ответов, отключения стенда и переключения риск/НДС.
- Если пользователь говорит «не вижу токены», проверять конкретный экран и
  конкретный блок: отдельный `Объёмы AI-нагрузки` не заменяет строку `Токены`
  в `Объёмы ресурсов` и не заменяет `Детализация → Сводка AI-метрик`. Для
  токенов нужен fallback из ответов раздела LLM, если legacy/сломанная формула
  token-ЭК дала 0 при положительной пользовательской нагрузке.
- Заполненный раздел `Объём токенов` является расчётным намерением, а не
  декоративной подсказкой. Если токенная нагрузка явно задана и положительна,
  Dashboard и Детализация обязаны показывать токены даже при stale/imported
  `ai_llm_used=false`; Health Check может предупреждать о противоречии, но UI
  не должен скрывать рассчитанную нагрузку прочерками.
- Для похожих дефектов нельзя проверять один избранный ЭК. Нужен e2e-контракт
  по всем положительным qty: каждая resource/AI-строка Dashboard с qty > 0
  должна быть видимой и непустой, а каждый ЭК с положительным qty должен иметь
  видимую непустую строку в `Детализация → Объём (qty)`. Операционные
  fallback-метрики AI должны покрывать не только `TOKENS`, но и `EMBEDDINGS`,
  и применять тот же capacity-множитель, что таблица qty.
- Выравнивание заголовков стендов проверять по фактическому режиму таблицы:
  если значения стендов выровнены вправо, названия стендов тоже должны быть
  выровнены вправо в основной Детализации и `Сводка AI-метрик`. Единицы
  измерения не выводить в заголовках и числовых ячейках стендов, если рядом
  уже есть `Ед.изм.`.
- `Сводка AI-метрик` в Детализации не должна жить отдельной произвольной
  сеткой: её стендовые колонки надо сверять по координатам с основной таблицей
  Детализации выше. Название метрики и единицу держать в разных столбцах:
  `Метрика` и `Ед.изм.`; заголовки `Ед.изм.` в основной таблице и AI-сводке
  должны иметь явное горизонтальное выравнивание.
- Для денежных AI-агрегатов проверять сходимость именно видимых чисел. Если
  строки ЭК в Детализации округлены до рублей, сводка должна суммировать тот
  же display-rounded слой по стендам и по итогам; расхождение даже на 1 ₽ в UI
  считается дефектом доверия.
- Risk-бары в Dashboard должны иметь одинаковый левый край и ширину, но этого
  мало: e2e обязан проверять отсутствие наложения бара/заливки на сумму. На
  узкой карточке сумму и процент переносить под бар, а не сжимать в одну
  строку до визуального столкновения.
- Для per-resource коэффициентов стендов проверять сопоставимость базы
  формулы. Если LOAD=120% от ПРОМ для HDD/RAM/SSD/S3, формула LOAD должна
  масштабировать ту же смысловую базу, что и ПРОМ, включая аддитивные части
  вроде холодной доли файлов. Иначе арифметически корректный множитель может
  дать логически слабее стенд.
- Для RAM не округлять промежуточный `CPU × коэффициент стенда` перед
  добавлением cache. Масштабируется полный ПРОМ-бейзлайн RAM:
  `ceil(PROМ vCPU) × RAM/vCPU + cache`, затем коэффициент ресурса. Пример
  пользовательского контроля: `56 ГБ × 1,2 = 67,2`, в целых ГБ к оплате
  показываем `68 ГБ`, а не `66 ГБ`.
