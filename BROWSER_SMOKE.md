# Browser Smoke Checklist

Ручной regression-чек для подтверждения текущей UX-модели. Запускать после **Ctrl+Shift+R** (hard-reload — иначе ESM-кэш браузера держит старые модули).

## Автоматический desktop suite

Для desktop-first проверки перед ручным проходом:

```bash
npm run smoke:desktop
npm run smoke:published   # после публикации релиза на GitHub Pages
```

Команда запускает Playwright на 1365×768 и параллельно проверяет Dashboard, Cost Optimization Planner, Decision Memo, Детализацию, Сравнение и Опросник. Помимо smoke-рендера, suite сверяет Dashboard/Details с production `calculate()`, проверяет пересчёт после изменения ответа, исключение disabled-стенда из totals, независимость риск-коэффициентов от НДС и реальные desktop user-flow клики: Quick Start, sidebar, period/stand controls, Опросник risk/VAT, Dashboard CTA планера, scenario tabs, активный JSON import/export/reset, bundle export/import/reset, legacy provider JSON VAT-policy import, Decision Memo `.md` download и PDF routing из шапки приложения. В Детализации suite отдельно проверяет, что пакетные единицы уведомлений отображаются как `тыс. SMS`, `тыс. писем`, `млн PUSH`, без строк вида `16 1000 SMS`, а PDF-режим Детализации через кнопку PDF, `Ctrl+Alt+P` и native `beforeprint` включает landscape `@page` и растягивает таблицы объёмов/стоимости на всю usable-ширину листа. Viewport guard отдельно проходит 1365×768, 1440×900 и 1920×1080: основные desktop-экраны не должны создавать document-level horizontal overflow, а app chrome не должен накладываться; Dashboard risk-card не должен раздувать `.dashboard-grid` min-content'ом. Visual-regression слой делает PNG-signal проверки пяти критичных экранов: скриншот должен быть не пустым/не однотонным, иметь ожидаемый desktop-размер, а основные chrome-блоки не должны перекрываться. Скриншоты сохраняются в `.playwright-mcp/`; временные отчёты Playwright тоже уходят туда через `playwright.config.js`.

Локально suite по умолчанию использует системный Chrome. В CI channel не фиксируется: GitHub Actions ставит bundled Chromium через `npx playwright install --with-deps chromium`, чтобы прогон не зависел от браузера на машине runner'а.

`npm run smoke:published` использует тот же Playwright config, но без локального `webServer`: скрипт ставит `PLAYWRIGHT_BASE_URL=https://smaltsev75-sudo.github.io/CALC_INFRA/` и гоняет короткий published-smoke. Это защищает от base-path регрессий GitHub Pages (`/CALC_INFRA/` вместо `/`). По умолчанию published smoke делает 1 retry (`PLAYWRIGHT_PUBLISHED_RETRIES=1`) и при сетевых ошибках пишет конкретный `HTTP <status>: <url>`, чтобы отличать CDN/Pages сбой от ошибки приложения.

В Детализации desktop-suite сверяет строки групп ЭК с production-моделью: порядок по `ИТОГО / год`, `ИТОГО / мес`, `ИТОГО / год` и долю группы в колонке `Доля, %`.

GitHub Pages публикуется workflow'ом `.github/workflows/pages.yml` из
`.pages-dist`, который собирается командой `npm run pages:build`. После
переключения Pages source на `GitHub Actions` legacy deploy-from-branch warning
GitHub больше не должен появляться в релизном цикле; проверка результата всё
равно остаётся `npm run smoke:published`.

На Node 26 upstream Playwright 1.60.0 может выводить `DEP0205 module.register()` warning при прямом `npx playwright ...`. Проектный `npm run smoke:desktop` запускает тот же CLI через `node --no-deprecation`, поэтому релизный прогон остаётся без шумных warning'ов.

Ручной checklist ниже остаётся нужен для сценариев, где важны UX-нюансы, клики и визуальная оценка, но базовые console/overflow regressions теперь должны ловиться автоматикой.

Для каждого пункта зафиксировано:
- **Путь** — куда кликать.
- **Ожидаемое** — что должно быть на экране.
- **Регрессия** — на что именно проверяем.

Если пункт не сошёлся — это баг, а не roadmap-вопрос. Скриншот + ID пункта (например, «13: видны дубли CTA в Качество расчёта»).

## A. Dashboard CTA dedup (Stage 17.3 / 17.4)

### A1. Карточка «Качество расчёта»
- **Путь**: открыть любой расчёт → Dashboard.
- **Ожидаемое**: ровно 2 кнопки — «Открыть проверку расчёта», «Уточнить данные».
- **Регрессия**: если видны «Допущения (N)», «Анализ чувствительности», «Сформировать memo» — **17.3 откатился**.

### A2. Карточка «Бюджет»
- **Путь**: тот же Dashboard.
- **Ожидаемое**: ровно 1 кнопка — «Подробнее».
- **Регрессия**: если видны «Сформировать memo» / «Рекомендованные действия» — **17.3 откатился**.

### A3. Блок «Следующие шаги» — без расширенного режима
- **Путь**: Sidebar → убедиться, что переключатель «Расширенные настройки» в footer **выключен** (серый). Создать расчёт с превышением бюджета (любой большой расчёт + установить `target_capex_rub` и `target_opex_monthly_rub` ниже текущего CAPEX/OPEX в Опроснике).
- **Ожидаемое**: в Next Steps видна подсказка «Изучить превышение бюджета», но НЕ видно «Найти драйверы стоимости».
- **Регрессия**: если «Найти драйверы стоимости» отображается без advanced — **17.4 откатился**.

### A4. Блок «Следующие шаги» — с расширенным режимом
- **Путь**: Sidebar → включить «Расширенные настройки». Тот же бюджет-overflow.
- **Ожидаемое**: в Next Steps теперь ДОЛЖНА появиться «Найти драйверы стоимости» (target = sensitivity_analysis).
- **Регрессия**: если её нет даже в advanced — гейт сломан (overgating).

## B. Основные пользовательские пути

### B1. Новый расчёт через «+ Расчёт»
- **Путь**: Sidebar → «Расчёты» → «+ Создать пустой расчёт».
- **Ожидаемое**: расчёт создан, открылся Опросник, видна шапка с именем по умолчанию.

### B2. Quick Start
- **Путь**: Sidebar → «Расчёты» → «Quick Start» (одна из preset-кнопок) → ответить на 7 макро-вопросов → создать.
- **Ожидаемое**: расчёт создан с `wizard`-meta, открылся Dashboard, видны цифры.

### B3. Опросник — Section accordion
- **Путь**: Опросник → клик по любой секции (свернуть/развернуть).
- **Ожидаемое**: секция меняет состояние, F5 переживает изменение.
- **Регрессия**: F5 сбрасывает — `questionnaireOpenSections` не persist'ится.

### B4. Опросник — `Перейти к полю` из Допущений
- **Путь**: Dashboard → блок «Готовность расчёта» (после 17.6) или Hero → бейдж «Допущения (N)» → одна из карточек → «Перейти к полю».
- **Ожидаемое**: открылся Опросник, страница прокрутилась к нужному полю, поле подсвечено glow'ом ~1.2с. Если поле в режиме «Не знаю» — фокус на кнопке «Не знаю» + snackbar «Нажмите "Не знаю", чтобы включить ручной ввод».
- **Регрессия**: остался на dashboard / поле не подсвечено / нет прокрутки.

### B5. Dashboard — переключение периода ₽/день·мес·год
- **Путь**: Dashboard → переключатель периода в Hero.
- **Ожидаемое**: ВСЕ значения (Hero / Distribution / Risk / Stand cards / AI metrics) меняются синхронно.
- **Регрессия**: какой-то блок не реагирует на period — не подключён к state.dashboardPeriod.

### B6. Dashboard — «С рисками / Без рисков» pill
- **Путь**: Опросник → «Параметры расчёта» → выключить мастер-toggle «Применять риск-коэффициенты» → Dashboard.
- **Ожидаемое**: pill в Hero меняется на «БЕЗ РИСКОВ» (orange), все суммы пересчитаны без накруток. 7 связанных полей в Опроснике становятся disabled.
- **Регрессия**: суммы не изменились / pill не переключился / поля остались enabled.

### B7. Импорт прайса JSON
- **Путь**: Sidebar → «Прайсы» (если есть в IA) → «Импорт прайса JSON» → выбрать файл provider.json.
- **Ожидаемое**: прайс применён, snackbar success, видна метка «Применён» рядом с провайдером.

### B8. История прайсов
- **Путь**: Sidebar → «Прайсы» → «История» (если есть).
- **Ожидаемое**: список до 3 предыдущих прайсов, кнопка «Откатить».

### B9. Прайс-бенчмарк
- **Путь**: Sidebar → «Прайсы» → «Бенчмарк» (если есть).
- **Ожидаемое**: сравнение текущего прайса с другими провайдерами для одинаковых ЭК.

### B10. Экспорт JSON
- **Путь**: header / sidebar → «Экспорт» → JSON.
- **Ожидаемое**: скачался файл `<calc-name>-<dd-mm-yyyy>.json`. Bundle с `BUNDLE_VERSION`.

### B11. PDF (Опросник)
- **Путь**: Опросник → «Печать в PDF».
- **Ожидаемое**: открылся PDF preview без visual artefact'ов. На dark-теме — текст не белый на белом.
- **Регрессия**: пустые блоки / sticky-thead overflow / тёмные фоны (см. PATCH 2.4.32 / 2.4.39).

### B12. Decision Memo
- **Путь**: Dashboard → Next Steps → «Сформировать обоснование расчёта» (или из Health Check).
- **Ожидаемое**: модалка с Markdown'ом, кнопки «Скопировать» / «Скачать .md».

## C. IA / Sidebar / advancedMode

### C1. Sidebar default state
- **Путь**: первая загрузка приложения.
- **Ожидаемое**: видны секции «Расчёты», «Мой расчёт», «Прайсы». **НЕ видны** «Элементы», «Вопросы» (admin-вкладки advanced-only).

### C2. Sidebar advanced state
- **Путь**: Sidebar footer → переключатель «Расширенные настройки» → ON.
- **Ожидаемое**: появляется секция «Администрирование» с «Элементы» + «Вопросы». F5 переживает.

### C3. Theme toggle
- **Путь**: header → иконка солнца/луны.
- **Ожидаемое**: тема переключилась мгновенно, без flash-of-wrong-theme. F5 переживает.

## D. Cost Optimization Planner (Stage 18.1.1 / 2.14.0)

Все пункты — на расчёте с **ненулевой стоимостью** (создать через Quick Start «Высокая нагрузка» либо открыть существующий тяжёлый расчёт). Без стоимости рычаги выдают saving 0 ₽ и часть проверок теряет смысл.

### D1. Карточка «План оптимизации стоимости» на Dashboard
- **Путь**: Dashboard → найти карточку «План оптимизации стоимости».
- **Ожидаемое**: одна карточка с одним CTA («Открыть план оптимизации» / equivalent). Не перегружена: нет старых 3 плиток, нет дублирующих кнопок «Применить план» / «Автооптимизировать» / «What-if» / «Scenario Pack» рядом.
- **Регрессия**: если видны 3 плитки или несколько CTA — Stage 18.1.1 откатился к ранней версии.

### D2. Открытие модалки и группировка рычагов
- **Путь**: Dashboard → «Открыть план оптимизации».
- **Ожидаемое**: модалка открывается быстро (без видимой задержки). Рычаги сгруппированы по **6 областям компромисса**, каждая группа — header с chevron + title + meta «N параметров · −X тыс. ₽ / мес · риск Y». Группы с правками или с доступными рычагами раскрыты по дефолту; blocked — свёрнуты или отображаются с lock-иконкой.
- **Регрессия**: плоский список рычагов без группировки → используется старый `buildEditableLevers` вместо `groupOptimizationLevers`.

### D3. Blocked-группа и кнопка «Разрешить ...»
- **Путь**: в Опроснике → «Параметры расчёта» → включить какое-нибудь constraint (например, «Защитить compliance» → запрещает снижение security). Вернуться в модалку планера.
- **Ожидаемое**: соответствующая группа отрисована со static header (без chevron, иконка `lock`), body виден всегда с причиной блокировки + inline-кнопка «Разрешить ...». Клик по «Разрешить ...» — снимает constraint (toggle на off), правки не применяются автоматически.
- **Регрессия**: blocked-группа становится clickable accordion / при клике на «Разрешить ...» меняются `settings/answers` помимо самого constraint.

### D4. Period switcher (день / мес / год)
- **Путь**: в модалке планера → секция «Итог» → segmented control периода.
- **Ожидаемое**: переключение меняет ТОЛЬКО отображение сумм в карточках Итога (Текущая / Экономия / После) и не трогает draft / per-lever значения. На daily — дробная часть `1` знак (например, `12.3 тыс. ₽ / день`).
- **Регрессия**: switcher не реагирует / меняет данные расчёта / sticks к `state.ui.dashboardPeriod` после закрытия (должен жить отдельно в `state.modals.costOptimizationPlanner.viewPeriod`).

### D5. Draft-flow (изменить → сбросить → применить → откатить)
- **Путь**: открыть планер → раскрыть любую non-blocked группу → изменить 1 рычаг (например, выбрать другой SLA или сместить slider буферов).
- **Ожидаемое**:
  1. карточка рычага показывает изменённое значение + saving обновился (per-group meta + общий Итог);
  2. кнопка «Сбросить параметр» возвращает рычаг к текущему значению расчёта;
  3. изменить 2–3 рычага в разных группах → нажать «Применить» → snackbar success, dashboard пересчитан, бейдж изменений в карточке планера сброшен;
  4. кнопка «Откатить последнее применение» возвращает `settings/answers/answersMeta` к snapshot'у до Apply.
- **Регрессия**: применение не персистится через F5 / откат не работает / saving считается странно (≠ разнице «До − После»).

### D6. High-risk SLA confirmation
- **Путь**: открыть планер → группа `reliability` → SLA-рычаг → выбрать значение, помеченное как high-risk (ниже текущего на 2+ уровня).
- **Ожидаемое**: inline-confirmation внутри карточки рычага («Это снизит надёжность на N уровней. Подтвердить?») + кнопка-подтверждение. До подтверждения — это значение в draft не уходит.
- **Регрессия**: значение применяется без confirmation / confirmation в модалке поверх модалки.

### D7. Modal scroll preserve
- **Путь**: открыть планер → проскроллить вниз (например, к 4-й группе) → изменить любой рычаг → раскрыть/свернуть accordion-группу.
- **Ожидаемое**: позиция scroll сохраняется — модалка не «прыгает» наверх при каждом state-изменении.
- **Регрессия**: scroll сбрасывается в 0 — [renderModals (js/ui/index.js)](js/ui/index.js) не выполняет preserve scroll-top через `overlay.dataset.modalName`.

### D8. Regression по Dashboard и смежным IA
- **Путь**: после закрытия планера — пройти Dashboard и Sidebar.
- **Ожидаемое**:
  - Карточка «Качество расчёта» — те же 2 кнопки из A1 (нет новых дублирующих CTA от планера).
  - Блок «Следующие шаги» — те же подсказки, что в A3/A4, не превращён в playbook планера.
  - Sidebar → «Прайсы» → «Импорт прайса» — только JSON, нет «Обновить с сервера» / fetch-варианта.
  - Sidebar default state (C1) — «Администрирование» скрыто; advanced (C2) — появляется.
- **Регрессия**: появились новые CTA в Health-card / Next Steps от планера; импорт прайса предлагает fetch; администрирование видно без advanced.

## Отчёт после прогона

Если все 27 пунктов сошлись — текущая UX-модель зафиксирована. Если нет — список расхождений (ID пункта + что именно не так + screenshot). Приоритет фиксов: A > B > D > C.

После любых правок IA/CTA/Dashboard/Planner — этот checklist прогнать заново.
