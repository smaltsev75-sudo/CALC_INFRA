# Руководство пользователя

Это руководство — для тех, кто **уже установил и запустил** калькулятор. Установка и системные требования — в [ReadMe.md](ReadMe.md). Архитектура и устройство кода — в [Architecture.md](Architecture.md). Регламент обновления цен и тестов — в [MAINTAINER_GUIDE.md](MAINTAINER_GUIDE.md).

**Целевой пользователь** — Product Owner и Архитектор: вы создаёте оценку TCO для собственного продукта или продукта команды, заполняете опросник, читаете результат, проверяете качество и формируете обоснование для бизнеса.

## Структура интерфейса

Слева — sidebar (сворачивается до 64 px на узких экранах). Сверху — TopBar с именем активного расчёта и быстрыми кнопками `Импорт JSON` / `Экспорт JSON` / `PDF` / `Сброс` / переключателем темы.

| Раздел | Назначение |
|---|---|
| **Расчёты** | Список расчётов: создать / открыть / дублировать / удалить / экспорт-импорт. На карточке — имя, дата, общая сумма, режим («С рисками» / «Без рисков»). |
| **Опросник** | ~80 вопросов о продукте + блок «Параметры расчёта» (буферы, риски, НДС, размеры стендов, AI-нагрузка). |
| **Дашборд** | Hero «Итого по расчёту», «Сводка состояния расчёта» (готовность / качество / бюджет / следующий шаг / оптимизация стоимости — один блок), распределение по категориям, вклад риск-коэффициентов, 5 стенд-карточек. |
| **Детализация** | Sub-tabs «Бюджет (₽)» / «Объём (qty)», построчно каждый ЭК × стенд + кнопка CSV. |
| **Сравнение расчётов** | Side-by-side 2–4 расчётов с дельтой по стендам и категориям. |

При включённом тогле **«Расширенные настройки»** (в нижней части бокового меню, рядом со «Справкой») появляется ещё одна группа — **Администрирование** с разделами «Элементы» (CRUD каталога) и «Вопросы» (CRUD опросника). По умолчанию скрыто. Состояние тогла переживает F5.

## Типовой сценарий использования

```
1. Расчёты → Новый расчёт (имя проекта)
        ↓
2. Quick Start (3 преcета) → быстрый старт с типовыми ответами
        ↓
3. Опросник → донастройка ответов (5–10 минут)
   • Пользователи / DAU / RPS
   • БД / файловое хранилище
   • Email / SMS / PUSH уведомления
   • Безопасность (152-ФЗ, ФСТЭК)
   • Пентесты, нагрузочное
   • AI / LLM / RAG (если нужны)
        ↓
4. Опросник → блок «Параметры расчёта»
   • Длительность фазы
   • Буферы (задачный + проектный)
   • Риск-коэффициенты
   • НДС / горизонт планирования
   • Размеры стендов (% от ПРОМ)
        ↓
5. Дашборд + Детализация — посмотреть результат
        ↓
6. Анализ расчёта — Качество / Допущения / Чувствительность / Бюджет
        ↓
7. Следующие шаги (блок на Дашборде) — что разумно сделать дальше
        ↓
8. Memo / PDF / JSON / CSV — поделиться результатом с командой
        ↓
9. Сравнение расчётов — несколько вариантов side-by-side
```

**Совет.** Если первый расчёт даёт «слишком дорого / слишком дёшево» — открой **Детализацию**, поищи аномальные строки. Часто это:
- забыли поставить `Q.peak_rps` (вся виртуалка считается от 0);
- лицензия СУБД на много vCPU при низком сценарии;
- разовые работы (внедрение / пентест) разнесены равномерно по `phaseDurationMonths` — увеличение фазы их «размазывает».

## Расчёты

### Создание расчёта

**Расчёты → Новый расчёт.** Имя — это имя проекта. Открывается мастер Quick Start с тремя пресетами:

| Пресет | Когда выбрать |
|---|---|
| **Стандартный** | Типовое B2B SaaS-приложение с умеренной нагрузкой. |
| **Высокая нагрузка** | Highload / e-com / fintech с пиковыми сценариями. |
| **Внутренний инструмент** | Корпоративное приложение для сотен пользователей. |

После выбора пресета — диалог уточнения (PD-152ФЗ / тип приложения / аудитория) и расчёт сразу переходит в Опросник, заполненный на ~70%. Дальше вы донастраиваете ответы под свой продукт.

### Сценарии внутри расчёта

Один расчёт может содержать несколько **сценариев** — альтернативные ветви ответов внутри той же модели. Сценарии нужны, когда вы хотите оценить **тот же продукт в нескольких конфигурациях**, не плодя десяток дублирующихся расчётов и сохраняя общие настройки (провайдер, НДС, риски) синхронно между ними.

**Расчёт vs сценарий — не путать.**

| | Расчёт | Сценарий |
|---|---|---|
| Уровень | Top-level проект (карточка в разделе «Расчёты») | Ветвь внутри расчёта |
| Файл JSON-экспорта | Один файл = один расчёт со всеми его сценариями | Самостоятельного файла нет |
| Имя в TopBar | Да (заголовок) | Нет (видно в табах ниже) |
| Когда нужен | Разные проекты / клиенты / продукты | Один продукт, разные конфигурации |

#### Как выглядит в интерфейсе

Между названием расчёта и кнопками `Импорт JSON / Экспорт JSON / PDF` живёт горизонтальная полоса вкладок:

```
[Базовый •] [MVP] [Pilot 100 пользователей ⋯]  [+ Сценарий]
```

- Активная вкладка подсвечена.
- Маркер «**•**» рядом с названием — в этом сценарии есть **ручные правки** (поля с бейджем «Вы изменили»). Tooltip на точке показывает количество.
- Кнопка `⋯` справа от названия — действия со сценарием: переименовать / дублировать / удалить.
- `+ Сценарий` справа — добавить новый.

Полоса вкладок видна **на всех экранах** (Опросник / Дашборд / Детализация / Сравнение). Переключение мгновенно меняет содержимое всего приложения — Дашборд и Детализация пересчитываются под ответы активного сценария.

#### Что общее, что персональное

Это важно понимать заранее, чтобы не удивляться, когда правка в одном сценарии «перетекла» в другой.

**Персональное (для каждого сценария — своё):**
- Ответы опросника (`Q.*` — пользователи, нагрузка, ИИ, безопасность и т.д.).
- Метки происхождения ответов (бейджи «Из профиля» / «Из масштаба» / «Вы изменили»).
- Профиль Quick Start (`wizard`), если был выбран при создании сценария.

**Общее (одно на весь расчёт — меняется во всех сценариях одновременно):**
- Глобальные настройки: НДС, провайдер, активный прайс, риск-коэффициенты, инфляция, сезонность, контingency, горизонт планирования.
- Размеры стендов (`standSizeRatio`), per-resource распределение нагрузки, AI-нагрузка на стендах (`aiStandFactor`).
- View-настройки: отключённые стенды, активная вкладка, развёрнутые секции опросника.
- Каталог ЭК и список вопросов (если вы их редактировали в Администрировании).

**Практический вывод.** Если вы хотите сравнить **«с НДС vs без НДС»** или **«Cloud.ru vs Yandex Cloud»** — это **разные расчёты**, не сценарии (переключатель НДС / провайдер — общие). Если же сравниваете **«MVP 100 пользователей vs Production 10 000 пользователей»** при том же провайдере и тех же рисках — это **сценарии**.

#### CRUD: создание, переключение, переименование, удаление

| Действие | Как сделать |
|---|---|
| **Создать сценарий** | Кнопка `+ Сценарий` в полосе вкладок. Открывается модалка переименования с пустым полем; впишите имя и нажмите Enter. **Новый сценарий наследует профиль Quick Start от активного**, но стартует с **пустыми ответами**. Семантика: «тот же продукт, другая конфигурация». Если нужна копия с ответами — используйте «Дублировать». |
| **Переключить активный** | Клик по вкладке. UI мгновенно пересчитывается. Snackbar: «Сценарий переключён». |
| **Переименовать** | `⋯` → «Переименовать». Имя обрезается до 60 символов. |
| **Дублировать** | `⋯` → «Дублировать». Открывается модалка с дефолтным именем `«<источник> (копия)»` — можно принять или ввести своё. Копируются и профиль Quick Start, и ответы, и метки происхождения. |
| **Удалить** | `⋯` → «Удалить». Confirm-модалка с явным предупреждением. **Последний сценарий удалить нельзя** — кнопка появится только при ≥ 2 сценариях. Глобальные настройки расчёта при удалении сценария **не теряются**. |

Все CRUD-операции пишутся в `localStorage` сразу же; индикатор персиста в TopBar мигает «Сохранено».

#### Примеры использования

**Пример 1 — Поэтапный запуск продукта.**
Команда планирует запустить продукт в три этапа и хочет видеть TCO каждой фазы при тех же риск-параметрах.

```
Расчёт «Мой продукт»
  ├─ Сценарий «Pilot 50 пользователей»
  │     DAU 30, RPS 5, БД 20 ГБ, без ИИ
  ├─ Сценарий «MVP 1 000 пользователей»
  │     DAU 400, RPS 80, БД 150 ГБ, RAG включён
  └─ Сценарий «Прод 10 000 пользователей»
        DAU 4 000, RPS 600, БД 2 ТБ, RAG + fine-tune
```

НДС, провайдер, инфляция, риск-коэффициенты — одинаковые во всех трёх. Переключаетесь между вкладками — видите TCO для каждой фазы. Сценарий «Сравнить сценарии» (см. ниже) положит их в одну таблицу.

**Пример 2 — Чувствительность к AI-нагрузке.**
Базовый сценарий без ИИ + два сценария с разной интенсивностью ИИ для оценки «сколько стоит AI-фича».

```
  ├─ Базовый — `ai_llm_used = false`
  ├─ AI лёгкая — `ai_users_share = 10 %`, `requests_per_user_day = 3`
  └─ AI тяжёлая — `ai_users_share = 80 %`, `requests_per_user_day = 30`
```

Сравнение между сценариями сразу показывает дельту по категориям AI и LICENSE, не трогая прочие категории.

**Пример 3 — Оптимизация одного риск-параметра.**
Базовый сценарий с реалистичным риск-набором + дубль с агрессивным сценарием понижения буферов.

```
  ├─ Реалистичный — buffer_task 30 %, buffer_project 20 %, contingency 15 %
  └─ Агрессивный  — buffer_task 10 %, buffer_project 10 %, contingency 5 %
```

⚠ **Важная оговорка для этого кейса.** Риск-коэффициенты — **глобальные настройки** (см. таблицу выше), они НЕ перенесутся между сценариями. Чтобы сравнить риск-настройки, удобнее использовать **отдельные расчёты** или **План оптимизации стоимости** (карточка на Дашборде), который явно работает с настройками рисков на одном calc'е.

Для **per-scenario** изменений подходят: количество пользователей, AI-параметры, политики безопасности, retention, частота резервного копирования — всё, что живёт в `Q.*`-ответах.

**Пример 4 — Заявка vs реальный масштаб.**
Заказчик сказал «1 000 пользователей», но архитектор подозревает, что через год будет 10 000. Делаем оба варианта рядом:

```
  ├─ Сценарий «Заявленный — 1 000 пользователей» (база Quick Start «Стандартный»)
  └─ Сценарий «Реалистичный — 10 000 + рост 50 %/год»
```

Получаем два TCO рядом и используем для аргументации с бизнесом. Дубликат с правкой пары полей делается за минуту.

#### Сравнение сценариев

При двух и более сценариях на Дашборде в блоке **«Следующие шаги»** появляется карточка **«Сравнить сценарии»** → открывает модалку «Сравнение сценариев»: каждая ваша конфигурация показана отдельной колонкой, по строкам — стоимость по стендам и категориям. Эта модалка также позволяет одновременно выбрать несколько провайдеров и увидеть cross-product `сценарии × провайдеры`.

Альтернатива без модалки — просто переключайте вкладки сценариев и сравнивайте Дашборд / Детализацию глазами. Часто этого достаточно.

#### Подводные камни и FAQ

- **Удалил активный сценарий — куда переключится?** На первый из оставшихся (`scenarios[0]`). Последний сценарий удалить нельзя.
- **Изменил настройку (НДС / провайдер) — она поменялась в других сценариях.** Это by design: общие настройки. Если хотите изолировать настройки — это **разные расчёты**, не сценарии.
- **При экспорте JSON попадает только активный сценарий?** Нет, в JSON-файле сохраняется **весь расчёт со всеми сценариями** + указатель на активный. При импорте всё восстановится.
- **Где видны «ручные правки» сценария?** Точка-индикатор на вкладке (только при ≥ 1 ручной правке). Сами поля помечены бейджем «Вы изменили» в Опроснике рядом с input'ом.
- **Re-apply профиля Quick Start затронет другие сценарии?** Нет, **только активный сценарий**. Каждый сценарий помнит свой `wizard`-снимок и свои manual-метки независимо.
- **Можно ли переименовать сценарий на пустую строку?** Нет — будет no-op, имя останется прежним. Длина имени ограничена 60 символами.
- **Что делать, если случайно удалил сценарий?** Только если есть бэкап JSON-файла — реимпортировать. Локальной истории удалённых сценариев нет.
- **На Дашборде вместо кнопки «Изменить параметры» вижу «Задать профиль сценария» — почему?** У этого сценария не задан профиль Quick Start (`wizard === null`). Это бывает для сценариев, созданных до v2.13.1 (раньше `+ Сценарий` создавал пустой профиль), либо при импорте JSON без профиля. Нажмите кнопку — откроется Quick Start с предзаполненными дефолтами, после submit профиль и derived-ответы применятся к этому сценарию (другие сценарии не затрагиваются). С v2.13.1 новые сценарии через `+ Сценарий` сразу наследуют профиль активного, поэтому empty-state встречается только для legacy.

### Что считается

```
costFinal = qty × pricePerUnit × billingIntervalMul × (риск-факторы) × НДС
```

где:
- `qty` — количество единиц для каждой пары (ЭК × стенд), вычисляется по формуле из ответов опросника.
- `pricePerUnit` — цена за единицу из каталога.
- `billingIntervalMul` — нормализация к месяцам (`daily / monthly / annual / oneTime`).
- Риск-факторы — буферы, инфляция, сезонность, сдвиг сроков, непредвиденные.
- НДС — независимая от рисков ось (включается отдельным переключателем).

Полная схема и список риск-коэффициентов — в [Architecture.md](Architecture.md).

### Режим «Без рисков»

Переключатель **«Учитывать риск-коэффициенты в бюджете»** в **Опросник → Параметры расчёта**. При выключении 7 связанных полей (буферы, инфляция, сезонность, расписание, непредвиденные) приглушаются. НДС применяется независимо.

В режиме «Без рисков»:
- На дашборде pill в Hero меняется `С РИСКАМИ` → `БЕЗ РИСКОВ` (зелёный → оранжевый).
- В карточке «Вклад риск-коэффициентов» показывается **потенциальная** наценка.
- В Детализации section-title «Стоимость, ₽» получает badge «Без рисков».

### Режимы НДС

Ставка НДС в РФ — параметр с историей (18% до 2019, 20% с 2019 по 2025, 22% с 01.01.2026). Калькулятор поддерживает три режима её определения (поле в **Опросник → Параметры расчёта → НДС**):

- **Авто (по дате)** — ставка берётся из встроенного справочника по дате создания расчёта. Дефолт для новых расчётов. Бейдж рядом с полем: «Авто 22% · дд.мм.гггг».
- **Вручную** — пользователь задаёт особую ставку (нерезидент, экспорт, льгота). Не пересчитывается автоматически. Бейдж: «Вручную 18%».
- **Заморожено** — ставка зафиксирована для согласованного бюджета. Обновления приложения не должны менять итог. Бейдж: «Заморожено 20%, дд.мм.гггг».

Переключение режимов — кнопки «Авто» / «Вручную» / «Заморозить» рядом с бейджем. Прямая правка поля «Ставка НДС, %» автоматически переводит расчёт в режим «Вручную».

**Предупреждение при пересечении даты изменения ставки.** Если ваш горизонт планирования пересекает дату, когда ставка НДС меняется (например 1 января 2026 года: 20% → 22%), приложение покажет non-blocking warning под полем НДС:

> Расчёт пересекает дату изменения НДС: 01.01.2026, 20% → 22%. Сейчас применяется ставка на дату расчёта: 20%. Для точной оценки разделите бюджет по периодам или задайте ставку вручную.

В текущей версии применяется одна ставка (на `vatEffectiveDate`); blended-режим (взвешенное усреднение по месяцам horizon) не реализован.

**Legacy-расчёты (созданные до 1 января 2026 года, обычно при ставке 20%).** При первом открытии в новой сессии вы увидите info-snackbar:

> Расчёт создан при ставке НДС 20%. Ставка зафиксирована, чтобы не изменить согласованные цифры. Сменить режим можно в Опроснике.

Это сделано специально: суммы старых согласованных бюджетов не меняются автоматически при обновлении приложения. Если нужно — переключите режим вручную в Опроснике.

**Где режим НДС виден:**
- В Опроснике — бейдж + 3 кнопки.
- В Decision Memo (раздел «Краткое резюме») — строка вида «Ставка НДС: 22% (авто, дата ставки: дд.мм.гггг)».
- В Сравнении — мелкий chip «НДС 22% · авто» под именем каждого расчёта; warning «Ставки НДС различаются — итоги не сопоставимы напрямую» при сравнении расчётов с разными rates.

### Стенды и инвариант «стенд ≤ ПРОМ»

| ID | Название | Дефолтный размер (% от ПРОМ) |
|---|---|---|
| DEV | DEV | 16% |
| IFT | ИФТ | 40% |
| PSI | ПСИ | 50% |
| LOAD | Нагрузка | 80% |
| PROD | ПРОМ | **100% (зафиксировано)** |

ПРОМ — эталон, к которому привязаны остальные стенды. Все три ratio-настройки (общий `standSizeRatio`, per-resource override 4×6, AI-нагрузка `aiStandFactor`) ограничены 0–100%. Раньше LOAD допускался до 120% («с запасом»), но это противоречило физическому смыслу нагрузочного стенда.

### Per-resource распределение нагрузки

В **Опросник → Параметры расчёта → Размеры стендов** есть таблица 4×6 (DEV/IFT/PSI/LOAD × CPU/GPU/RAM/SSD/HDD/S3). Это позволяет, например, для DEV установить CPU=20%, RAM=10%, SSD=5% — стенды получат разную долю каждого ресурса вместо единого процента.

Item-ы без поля `dashboardResource` (Услуги, Лицензии, Безопасность, Трафик) используют общий `standSizeRatio.<STAND>`.

### AI-нагрузка на стендах

Master-toggle `Q.ai_agent_mode` + per-stand `aiStandFactor`:

| Стенд | Дефолт |
|---|---|
| DEV | 2% |
| ИФТ | 20% |
| ПСИ | 50% |
| Нагрузка | 100% |
| ПРОМ | 100% (заперт) |

DEV ≠ 0 — это разработческий AI-traffic (регрессионные тесты, ad-hoc разработка, демо). Симметрично всем AI-метрикам: токены, эмбеддинги, agent-cpu, memory-storage.

## Анализ расчёта

Анализ расчёта — это группа инструментов, которые помогают понять, насколько ваш расчёт «хороший» и где тонкие места. Все они доступны с Дашборда.

### Качество расчёта (Health Check)

**Дашборд → карточка «Качество расчёта» → Открыть проверку расчёта →**. Карточка показывает только score-state и две прямые кнопки: «Открыть проверку расчёта» и «Уточнить данные» (мастер дозаполнения). Переходы к допущениям, чувствительности и memo вынесены в блок «Следующие шаги», чтобы один path к одному действию (Stage 17.3 IA-rule).

Показывает оценку 0–100 на основе:
- доли заполненных вопросов;
- количества рискованных допущений «по умолчанию»;
- количества ошибок в ответах (несовместимые сочетания, отрицательные значения).

Внутри модалки — список конкретных warnings и errors с переходом на источник (на конкретное поле в Опроснике).

### Динамика качества

В той же модалке — секция **«Динамика качества»**: timeline последних 20 точек, дельта «было → стало», лучшее значение. Snapshot записывается:
- при открытии Health Check;
- при завершении шага в Мастере уточнения;
- при открытии действия из «Следующих шагов».

Хранится локально (`localStorage`), при экспорте JSON-файла **не передаётся** — это ваша приватная история действий.

### Реестр допущений

**Дашборд → бейдж «Допущения (N)» в Hero**.

Список ответов, на которые вы не дали явного значения и калькулятор использовал defaultIfUnknown. Помогает увидеть «сколько у меня сейчас догадок и какие из них рискованные» (рискованные подсвечены).

### План оптимизации стоимости

**Дашборд → «Сводка состояния расчёта» → секция «Оптимизация стоимости» → `Открыть план оптимизации`**.

Инструмент для PO + Архитектора: показывает, **за счёт чего можно снизить стоимость и какой ценой**, и позволяет применить выбранные правки к расчёту одним явным действием.

После Stage 18.2.x отдельная Dashboard-карточка `План оптимизации стоимости` удалена — точка входа встроена внутри «Сводки состояния расчёта» как secondary-action. Вся работа происходит в модалке. Если бюджет превышен и primary `Следующий шаг` уже ведёт в planner, дублирующая teaser-кнопка скрывается — кнопка остаётся одна (Stage 17.3 контракт «один target = один CTA»).

#### Что важно знать сразу

- **Это не «автооптимизатор» и не «применить план целиком».** Вы редактируете конкретные параметры (значения слайдеров, дней retention, % буфера, SLA), и только после нажатия `Применить изменения` они уходят в расчёт.
- **До явного Apply активный расчёт не меняется.** Открытие модалки, переключение уровня, toggle ограничений, правка значений — всё это живёт в **черновике** (draft). Дашборд / Детализация / PDF продолжают показывать прежний расчёт.
- **Черновик переживает закрытие модалки**, но **теряется при F5** (runtime-only, без localStorage). Это сделано осознанно: оптимизация — это «сессия принятия решения», не персистентная сущность.
- **Rollback тоже session-only.** После применения в модалке появляется ярлык «Откатить последнее применение» — он восстанавливает состояние, какое было до Apply, пока модалка открыта. F5 либо новое применение перетрут snapshot.

#### Структура модалки

1. **Уровень оптимизации** — три вкладки сверху:
   - **Консервативный** (0–5 %) — минимальная боль, без SLA, без AI/RAG.
   - **Амбициозный** (5–15 %) — управляемый компромисс, AI/RAG разрешён, SLA остаётся opt-in.
   - **Экстремальный** (15–25 %) — пересмотр требований, AI/RAG разрешён, SLA остаётся opt-in.

   Дефолт — **Амбициозный**. Уровень задаёт три вещи: целевой диапазон экономии, агрессивность подсказок «Рекомендуется» и **стартовый набор ограничений**.

2. **Ограничения** — шесть toggle'ов («Можно снижать SLA», «Можно уменьшать стенды», «Можно снижать риск-буферы», «Можно уменьшать AI/RAG», «Можно уменьшать retention», «Защитить compliance»). Toggle фильтрует доступные рычаги.

   **Семантика «touched/defaults» (гибрид):** при первом выборе уровня применяются его дефолтные ограничения. Как только вы **явно** нажали toggle, дальнейшая смена уровня **не перетрёт** ваш выбор — ваше «можно снижать SLA» останется, даже если переключиться на Консервативный. Не тронутые ограничения по-прежнему берутся от уровня. Маркер «·» рядом с подписью показывает «вы явно тронули этот toggle».

3. **Итог** — три карточки (текущая стоимость / после изменений / экономия) + строка статуса + **переключатель периода** (день / месяц / год):
   - «Изменений пока нет.»
   - «Попадает в диапазон 5–15 %.»
   - «Пока 3.8 % — ниже цели 5–15 %. Добавьте рычаги.»
   - «11.1 % выше верхней границы 15 %. Возможно, выбранные изменения ближе к следующему уровню.»
   - «Не удалось пересчитать черновик. Проверьте значения.»

   Все числа пересчитываются через **clone + calculate** (не оценочно). Базовая стоимость берётся из текущего активного расчёта. **Суммы всегда показываются в тыс. ₽** для согласованности с дашбордом и снижения визуального шума. Period switcher на первом открытии модалки берёт текущий период дашборда; дальше живёт независимо в state модалки.

4. **Рычаги** — сгруппированы по **областям компромисса** (что вам придётся уступить):
   - **Инфраструктура и стенды** — `allowNonProdReduction` гейтит DEV / ИФТ / ПСИ / нагрузочный.
   - **Надёжность и SLA** — `allowReliabilityTradeoff`.
   - **Данные и хранение** — `allowRetentionReduction` для backup / log retention.
   - **AI / RAG** — `allowAiReduction` для output-токенов, корпуса, эмбеддингов.
   - **Риски и резервы** — `allowRiskBufferReduction` для bufferTask / bufferProject / kContingency / kScheduleShift.
   - **Горизонт планирования** — без constraint, гейтится только `appliesIf horizon > 3` и skipInTiers Консервативный.

   Каждая группа — **аккордеон** с header'ом, показывающим summary: количество доступных параметров, накопленную экономию по группе (`−X тыс. ₽ / мес`), максимальный риск внутри группы. Экономия по группе считается **точно** — отдельный clone + calculate, где применены только изменения этой группы (а не маржинальная эвристика).

   **Заблокированные группы** (constraint=off) **не исчезают** — header показывает `lock` icon, тело видно с причиной «Заблокировано: ...» и inline-кнопкой `Разрешить ...`. Кнопка только снимает constraint (не применяет правки) — это короткий путь без скролла к блоку «Ограничения» наверху.

   **Пустые группы** (constraint=on, но `appliesIf` отсеял все spec'и — например AI-группа без `ai_llm_used=true`) сортируются в самый низ списка и показывают «Нет применимых параметров для текущего расчёта».

   **Состояние accordion** (раскрытые/свёрнутые группы) — runtime-only в state модалки, **не переживает F5** (как и черновик). По дефолту раскрыты группы с правками или с доступными рычагами; blocked — свёрнуты.

   Каждый рычаг внутри группы — это карточка с:
   - заголовком и risk-badge (Низкий / Средний / Высокий риск);
   - редактируемым контролом — `<select>` для enum (SLA, retention), `<input type="number">` для percent / int / float;
   - текущим базовым значением «Сейчас: X»;
   - подсказкой «Рекомендуется для уровня: Y» (если ваше значение отличается от рекомендации);
   - диапазоном допустимого — слайдер не может **увеличить** значение выше исходного (это инструмент оптимизации, не повышения);
   - описанием последствия;
   - кнопками `Сбросить параметр` (убрать одну правку из черновика) и `Перейти к полю` (только для answer-полей — открыть соответствующий вопрос Опросника).

5. **Footer**:
   - `Сбросить изменения` — очищает все правки в черновике (level и ограничения остаются).
   - `Применить изменения` — применяет правки к расчёту. **Активна только** при ≥ 1 валидной правке и при отсутствии ошибок в preview.
   - `Закрыть` — закрывает модалку. Черновик сохранится до F5.

#### Inline high-risk confirmation

Если среди правок есть **high-risk** изменения (снижение SLA, снижение `kContingency` — резерв на риски, снижение `kScheduleShift` — запас на сдвиг расписания), нажатие `Применить изменения` **не применяет их сразу**. Вместо этого открывается inline-панель (внутри той же модалки, без отдельного диалога):

```
Подтверждение применения изменений с высоким риском

Вы применяете изменения, которые могут заметно повлиять на надёжность или резервы:
  • Снизить целевой SLA  99.95 → 99.5
    Последствие: Выше допустимое время простоя; ниже требования к резервированию.
  • Снизить непредвиденные расходы  0.05 → 0.02
    Последствие: Меньше подушка на непредвиденное.

[Подтвердить изменения]  [Отмена]
```

- `Подтвердить изменения` запускает реальный apply.
- `Отмена` снимает подтверждение; черновик не трогается.

#### Rollback (откат последнего применения)

После успешного Apply в верхней части модалки появляется ярлык:

```
Последнее применение можно откатить, пока модалка открыта.  [Откатить последнее применение]
```

Откат восстанавливает `settings / answers / answersMeta` до состояния перед последним Apply (snapshot хранится в `state.modals.costOptimizationPlanner.lastApplySnapshot`) и пересчитывает дашборд. Черновик пересоздаётся свежим на восстановленном расчёте, уровень и тронутые ограничения сохраняются.

Snapshot теряется в трёх случаях:
1. F5 (runtime-only).
2. Закрытие модалки → следующее открытие → следующий Apply (предыдущий snapshot перезаписан).
3. Ручное действие в Опроснике или другом инструменте, мутирующее calc (snapshot всё ещё доступен через ярлык, но восстановит **только** те поля, которые менял Apply — изменения других полей сохранятся).

#### Доступные рычаги MVP

| Группа | Поле | Условия |
|---|---|---|
| Инфраструктура и стенды | `standSizeRatio.LOAD / PSI / IFT / DEV` | `allowNonProdReduction` |
| Риски и резервы | `bufferTask`, `bufferProject`, `kContingency`, `kScheduleShift` | `allowRiskBufferReduction` |
| Горизонт планирования | `planningHorizonYears` (≥ 3) | НЕ в Консервативном |
| Надёжность и SLA | `sla_target` | `allowReliabilityTradeoff` И НЕ в Консервативном; **высокий риск** |
| Данные и хранение | `backup_retention_days` | `allowRetentionReduction`; floor 90 дней при `protectCompliance` |
| AI / RAG | `ai_avg_output_tokens` | `ai_llm_used=true` И `allowAiReduction` |
| AI / RAG | `rag_corpus_size_gb`, `rag_embeddings_million` | `rag_needed=true` И `allowAiReduction` И НЕ в Консервативном |

SLA-значения берутся **из вопроса опросника** `sla_target` (не хардкод) — если вопрос отсутствует / без options, рычаг SLA не появится.

#### FAQ

- **Где увидеть, что я редактирую: расчёт или черновик?** В шапке модалки — disclaimer «Изменения сохраняются в черновике и применяются к расчёту только после нажатия „Применить изменения“». Пока вы не нажали Apply, дашборд / Детализация / PDF показывают **прежнее** состояние.
- **Можно ли применить только часть правок?** Apply применяет все правки в черновике одновременно. Если нужно применить выборочно — сначала снимите ненужные через `Сбросить параметр` на конкретных рычагах, а потом нажмите Apply.
- **Можно ли применить план дважды?** Нет «плана» как монолита, есть набор ваших правок. После Apply черновик очищается (правки уже применены); вы можете внести новые и применить ещё раз. Snapshot для отката будет перезаписан.
- **Применяет ли rollback правки обратно в черновик?** Нет — rollback восстанавливает **расчёт**, не черновик. Черновик после rollback — пустой (правок нет, можно начать заново).
- **Что делать, если Apply сделал что-то не то, а F5 уже нажат?** Snapshot потерян. Восстановить можно либо вручную (зная исходные значения), либо из JSON-бэкапа, если он есть. План оптимизации — инструмент для **осознанных** правок; для дорогостоящих экспериментов лучше делать JSON-экспорт перед Apply.

### Сводка состояния расчёта

**Дашборд → блок «Сводка состояния расчёта» (сразу под Hero)**.

После Stage 18.2 четыре отдельных блока (Готовность / Качество / Бюджет / Следующие шаги) объединены в один композитный status-блок. Он отвечает на 3 управленческих вопроса:

1. **Можно ли идти к обсуждению?** — главный badge:
   - **Готов к обсуждению** (зелёный) — блокеров нет.
   - **Требует уточнения** (жёлтый) — есть несрочные замечания (бюджет не задан, рискованные допущения, устаревший прайс).
   - **Есть блокеры** (красный) — есть критичные ошибки (`health_errors`), низкое качество (`score < 60`) или расчёт пуст.

2. **Есть ли проблемы по качеству или бюджету?** — две диагностические строки в одну линию:
   - **Качество расчёта** — «Критичных проблем не найдено» / «Есть N критичных проблем» + CTA «Открыть проверку».
   - **Бюджет** — «CAPEX и OPEX в пределах бюджета» / «CAPEX превышает на +X %» / «Целевой бюджет не задан» + CTA «Посмотреть рекомендации» (или «Указать бюджет», если target_*_rub не заполнены).

3. **Что делать следующим шагом?** — embedded primary action (один). Берётся первый из `getActiveNextSteps()` (сортировка по severity → top action). Кнопка ведёт в тот же инструмент, что и раньше: Health Check / Реестр допущений / Анализ чувствительности (advanced) / Бюджетные ограничения / Decision Memo / Мастер уточнения / Импорт прайса / Сравнение сценариев.

Не дублирует Health Check: модалка `Качество расчёта` (диагностический отчёт по 30+ полям) и сама composite-сводка — это два разных уровня детализации. Сводка — short-form status; модалка — полный список findings.

### Анализ чувствительности (только в расширенном режиме)

**Sidebar → переключатель «Расширенные настройки» → Дашборд → блок «Следующие шаги» → подсказка «Найти драйверы стоимости»**.

Основной анализ стоимости находится на Дашборде:
- **Распределение по категориям** — структура затрат (что сейчас дороже всего: Услуги / Лицензии / Аппаратные ресурсы / Безопасность / Трафик).
- **Вклад риск-коэффициентов** — сколько добавила наценка (буферы / инфляция / резерв / сдвиг расписания) и в чём её источник.

Для большинства задач этих двух блоков достаточно — они прямо отвечают на вопросы «что дорого?» и «сколько добавили риски?».

Анализ чувствительности — расширенный perturbation-инструмент для архитектора: для каждого числового параметра он временно изменяет его на ±10 % (boolean — переключает) и пересчитывает итог, ранжируя поля по abs-дельте. Отвечает на другой вопрос: **«какую вводную выгоднее изменить, чтобы снизить итог»**. Категория может быть огромной, но неэластичной (фиксированный SKU прайса) — sensitivity находит маленькие, но эластичные рычаги.

Доступен только при включённом «Расширенном режиме» — иначе подсказка «Найти драйверы стоимости» в «Следующих шагах» скрыта (Stage 17.4 IA-rule).

### Бюджетные ограничения

**Дашборд → Hero «Бюджет» → клик**.

Сравнение CAPEX и OPEX вашего расчёта с целевыми значениями (`target_capex_rub`, `target_opex_monthly_rub` из Опросника). Показывает превышение / запас, рекомендации.

### Decision Memo

**Дашборд → блок «Следующие шаги» → «Сформировать обоснование расчёта»** (также доступно из Health Check и Бюджетных ограничений).

Собирает управленческое обоснование расчёта в формате Markdown:
- название расчёта, активный сценарий, провайдер и версия прайса;
- итоговые CAPEX и OPEX;
- оценка качества из Health Check;
- источники ответов, рискованные допущения;
- ошибки и предупреждения;
- top-5 драйверов стоимости;
- статус относительно бюджета;
- рекомендации.

**Действия**: «Скопировать Markdown» (в буфер обмена) или «Скачать .md». Memo собирается на лету, ничего не сохраняется и не отправляется наружу.

## Следующий шаг (внутри «Сводки состояния расчёта»)

После Stage 18.2 отдельной карточки «Следующие шаги» нет — её заменил **embedded primary action** внутри «Сводки состояния расчёта». Это сделано осознанно: четыре дашборд-карточки, описывающие одно и то же состояние, конкурировали между собой и заставляли собирать общую картину из разрозненных блоков.

**Это не список изменений, которые приложение применит автоматически.** Кнопка «Следующего шага» открывает существующий инструмент — Health Check, Реестр допущений, Анализ чувствительности, Бюджетные ограничения, Memo, Мастер уточнения, Импорт прайса, Сравнение расчётов. Никакого apply, preview, rollback, мутации сценария.

### Какой действие подскажет блок

Берётся первое из `getActiveNextSteps()`, отсортированных по severity:

- **Уточнить критичные вводные** — Мастер уточнения, если в расчёте есть error-findings или score < 60.
- **Просмотреть предупреждения** — Health Check при ≥ 3 warnings.
- **Проверить рискованные допущения** — Реестр допущений при ≥ 3 рискованных значениях по умолчанию.
- **Изучить превышение бюджета** — Бюджетные ограничения при превышении CAPEX/OPEX. Сопутствующая подсказка **«Найти драйверы стоимости»** (Анализ чувствительности) появляется только в расширенном режиме (Stage 17.4).
- **Обновить прайс провайдера** — Импорт прайса, если применённый прайс заметно отстаёт от поставляемого с приложением.
- **Сравнить сценарии** — Сравнение сценариев при ≥ 2 сценариях.
- **Сформировать обоснование расчёта** — Decision Memo, когда расчёт выглядит готовым.

Полный список рекомендаций (если их больше одной) доступен из тех же инструментов — Health-модалка и Реестр допущений показывают свой набор подсказок, Бюджетные ограничения — свой. Дашборд показывает только **главное следующее действие**, чтобы не дробить экран.

### Severity и сортировка

- **Критично** — красная полоса (errors / превышение бюджета).
- **Важно** — оранжевая (low score / risky assumptions / устаревший прайс).
- **Полезно** — зелёная (множество сценариев — есть что сравнить).
- **Информационно** — серая (memo как low-priority default).

На один target — один CTA на Dashboard (дедуп, Stage 17.3 / 18.2).

## Прайс

Прайс — это таблица цен ЭК для активного провайдера. Обновляется только через **Импорт прайса JSON**. Автоматического обновления с сайтов провайдеров нет (см. ниже).

### Импорт прайса JSON

**Опросник → блок «Тарифы провайдера» → Импорт прайса**.

Файл-picker открывает мастер импорта. Поддерживаемые форматы:
- **Provider-JSON** со `schemaVersion: 1` — мастер пропускает mapping и сразу проверяет.
- **CSV** (запятая, точка с запятой, табуляция; UTF-8 с BOM или без). Распознаются заголовки на русском и английском.
- **JSON-массив** объектов: `[{ "service": "vCPU shared", "price": 920, "unit": "month" }, ...]`.

**Шаги мастера**:

1. Выбрать провайдера (по умолчанию — текущий из настроек расчёта) и нажать «Выбрать CSV/JSON файл».
2. **Превью** первых 8 строк с авто-маппингом.
3. **Сопоставление** — каждая строка → выпадающий список ЭК. Уровни уверенности:
   - **высокая** — точное совпадение id или известный синоним.
   - **средняя** — пересечение слов между названием в файле и названием ЭК ≥ 50%.
   - **низкая** — пересечение слов 30–50% (требует подтверждения).
   - **нет** — авто-маппинг не предложен; нужно выбрать вручную.
4. **Проверка** — сводка mapped / unmapped / errors. Кнопка **Применить прайс** активна только если ошибок нет.

**Что мастер проверяет**:
- Цена — финитное положительное число (`14 400,50` → 14400.5).
- Каждая строка сопоставлена с существующим ЭК.
- Нет двух строк, сопоставленных с одним и тем же ЭК.
- В файле не более 1000 строк (защита от случайной загрузки 10 МБ-CSV).

Применённый прайс уходит в override pipeline — в historyStack попадает прежний override, можно откатить через «Откатить на прайс <ver>».

### Жизненный цикл override'а

1. Применяете новый прайс — старый снимок уходит в **history-стек** (лимит 3).
2. Каждый расчёт хранит `providerVersion` — маркер «применил какой override».
3. Если в браузере появляется более свежий override, чем у активного расчёта — над ним появляется индикатор **«Старый прайс»** + кнопка **«Пересчитать на новом прайсе»**.
4. Применение — атомарное: либо все ЭК расчёта получают новые цены, либо ничего.

### История прайсов

**Опросник → блок «Тарифы провайдера» → История**.

Модалка показывает:
- **Current** — текущий applied override (источник, версия, время применения).
- **Previous 1..3** — последние снимки в history-стеке.

Для каждой записи:
- Версия и время применения.
- Top-3 `↑` (что подорожало) и `↓` (что подешевело) — delta-pill с tooltip «pricePerUnit X → Y ₽».
- Кнопка **Откатить на эту точку**.

Откат на точку N работает по семантике `git reset --hard`: все более новые точки удаляются вместе с current; target становится current; точки старше target сохраняются как «прошлое».

### Прайс-бенчмарк

**Опросник → блок «Тарифы провайдера» → Прайс-бенчмарк**.

Read-only сравнение цен провайдеров: таблица 4 категории (CPU / RAM / STORAGE / NETWORK) × все провайдеры. Каждая ячейка — представительная цена категории (key-item: `cpu-vcpu-shared` для CPU, `ram-gb` для RAM, и т. д.).

В каждой категории самый дорогой ЭК подсвечен (`top-expensive highlight`). Помогает быстро увидеть «у кого дороже за единицу». **Не путайте с Анализом чувствительности**: top-expensive — это просто цена за единицу, чувствительность — влияние на итоговый бюджет.

Сортировка: клик на header категории → asc/desc по этой колонке. По умолчанию — по «Итого» asc.

Бейдж delta-pill показывается, если effective price отличается от bundled-default (т. е. провайдер имеет применённый override).

### Сводка по провайдеру (in-questionnaire)

Под блоком выбора провайдера в Опроснике — сводка по 6 категориям активного провайдера: CPU / RAM / Storage / Network / Licenses / Services. Для каждой категории:
- название (с иконкой);
- список ЭК с ценой `pricePerUnit ₽`;
- top-expensive ЭК подсвечен;
- если есть applied override — delta-pill рядом с ценой.

### Почему нет автоматического обновления

Калькулятор работает строго локально и не делает сетевых запросов к сайтам провайдеров. Все цены поставляются вместе с приложением в виде JSON-файлов. Кратко — это сделано из соображений безопасности (CSP `connect-src 'self'`), отсутствия публичных billing-API и хрупкости HTML-парсинга. Полное обоснование и регламент квартального обновления цен — в [MAINTAINER_GUIDE.md → Provider Price Update Workflow](MAINTAINER_GUIDE.md#provider-price-update-workflow).

### Элементы

CRUD каталога ЭК (36 позиций, включая 7 AI-позиций):
- Создать / редактировать / удалить элемент.
- Поправить цену, единицу, категорию, формулу qty.
- JSON-импорт / экспорт каталога.
- **Цены CSV** (экспорт / импорт) — массовое обновление цен по `id`. Подробности формата: см. ниже.

#### Цены CSV (admin)

Формат:
```
id;name;vendor;unit;category;resourceClass;billingInterval;pricePerUnit;priceUpdatedAt;priceSource
```

Импорт принимает любой CSV с минимум двумя колонками `id` и `pricePerUnit`. Авто-детект разделителя (`;` или `,`). Десятичный — `.` или `,`. Match по id, обновляется только `pricePerUnit`.

Защита от опечаток:
- **Корректность** — число, не NaN, не Infinity.
- **Граничные** — 0 ≤ цена ≤ 10¹².
- **Аномалии** — если новая цена ≥ 10× старой ИЛИ ≤ старая/10, в отчёте предупреждение «возможна опечатка».

## Troubleshooting

### Браузер кэширует старую версию

После обновления приложения вы видите старые названия ЭК / тексты / интерфейс?

**Решение**: жёсткий reload — **Ctrl+Shift+R** (Windows/Linux) или **Cmd+Shift+R** (macOS). Браузер агрессивно кэширует ES-модули; обычный F5 не всегда очищает.

### CSV/JSON не загружается — «Файл не является корректным JSON»

Проверьте кодировку (должно быть UTF-8) и парность скобок. Самая частая ошибка — экспорт из Excel с ANSI / Windows-1251 кодировкой. Откройте файл в Notepad++ → Кодировка → Преобразовать в UTF-8 → сохранить.

### CSV-импорт цен помечает все строки как аномалии

Возможно, неверный десятичный разделитель — приложение видит `1,500` (рус.) как 1500 рублей вместо 1.5. Используйте экспорт текущих цен из приложения как **шаблон** — оно генерирует CSV с корректным форматом.

### Расчёт даёт 0 ₽ по всему

Скорее всего, не заполнен `peak_rps` или `microservices_count` или `db_count` — большинство ЭК зависят от этих параметров. При нулях формула qty тоже даёт 0.

**Решение**: пройдитесь по Опроснику и заполните 5–7 ключевых нагрузочных параметров.

### Tooltip пропадает / не появляется

Tooltip = `title=` атрибут. Появляется после 1–2 секунд hover'а. Это поведение браузера, мы не контролируем задержку.

Для info-кнопки (`<button>` с иконкой `?` или `i`) — клик открывает модалку с полным описанием.

### Данные пропали после закрытия браузера

Причины:
- **Режим инкогнито/приватный** — localStorage очищается при закрытии. Используйте обычное окно.
- **Очистка cookies/site-data** — браузер удалил вместе с другими данными. Восстановление возможно только из JSON-бэкапа.

Профилактика: периодически делайте **Расчёты → Полный экспорт**, особенно перед обновлением приложения или сменой браузера.

### «Прайс обновляется в другой вкладке»

Cross-tab защита: вы открыли несколько вкладок калькулятора, и в одной из них уже идёт применение того же провайдера. Подождите, пока операция там завершится, либо закройте ту вкладку. TTL lock 60 секунд истекает сам.

### Опросник: длинная анкета, сложно ориентироваться

- Используйте **аккордеоны секций** (раскрыть/свернуть слева от названия).
- **Прогресс-чип** в заголовке каждой секции показывает «N / M уточнено».
- **Подгруппы** (RAG / LLM / PDn / Customization) тоже сворачиваемые.
- Состояние «что раскрыто» persist'ится в localStorage — после F5 вернётесь туда же.

### Поле disabled, не могу изменить

Это **зависимое от master-toggle поле** (`dependsOn`). Над/рядом с ним должен быть переключатель, который его открывает. Tooltip объясняет: «Поле неактивно: сначала включите …».

Например, поля RAG-параметров активируются после включения мастера `Q.rag_needed`.

### Поле зачёркнуто в опроснике

Это `disabled` или приглушённое — обычно потому, что включён режим «Без рисков». Параметры рисков (буферы, инфляция, сезонность, расписание, непредвиденные) тогда не применяются и приглушаются.

## Импорт прайсов провайдера и НДС (с версии 2.16.0)

С версии 2.16.0 (Stage VAT-2) калькулятор различает цены **с НДС** и **без НДС** на уровне JSON-файла провайдера.

### Что важно

- **Внутри расчёта цены всегда хранятся без НДС.** НДС применяется отдельно по настройкам расчёта (Опросник → блок «НДС»). Это значит, что итог НЕ может оказаться завышенным из-за двойного учёта НДС.
- В блоке «Тарифы активного провайдера» появилась поясняющая метка: «Цены сохранены без НДС. НДС применяется отдельно в расчёте.»
- Если источник прайса содержал цены С НДС — это видно в дополнительной строке: «Источник содержал цены с НДС. При импорте они сохранены как цены без НДС.»

### Импорт нового JSON

Если вы импортируете провайдерский JSON современного формата (с явной политикой НДС в файле), приложение применяет цены автоматически.

Если вы импортируете старый JSON-файл (без явного указания политики НДС), приложение спросит:
- **Цены без НДС** — цены применяются как есть (net);
- **Цены с НДС 20%** — цены делятся на 1.20 (период 2019–2025);
- **Цены с НДС 22%** — цены делятся на 1.22 (с 01.01.2026);
- **Отмена** — импорт не выполняется, прайс не меняется.

По умолчанию вариант **не выбран** — вы должны явно указать, что в файле, чтобы избежать ошибки. Текстовая пометка «с НДС» в описании прайса (priceSource) не используется как подсказка — выбор всегда делает пользователь.

### Старые сохранённые расчёты

При открытии расчёта, созданного до версии 2.16.0, может появиться напоминание: «Старые расчёты могли учитывать НДС дважды. Проверьте применённый прайс…»

Расчёт **не меняется автоматически**. Если хотите пересчитать на актуальном прайсе:
1. Откройте блок «Тарифы активного провайдера» (CTA «Перейти к тарифам» в напоминании).
2. Импортируйте JSON провайдера (file-picker) — современный формат применится без вопросов; старый — попросит выбрать политику НДС.
3. Нажмите «Пересчитать на новом прайсе» — текущий расчёт обновится с новыми ценами и тем НДС, что задан в Опроснике.

### Если провайдер — VK Cloud

Цены VK Cloud в текущем bundled-комплекте помечены как **«НДС-политика принята по допущению»**. Это означает, что цены — realistic-stub (не верифицированный публичный прайс), и они трактуются как gross 22%. Когда появится верифицированный источник, метка обновится автоматически при следующем `npm run generate:providers`.
